# Техническое задание: облачный сервис «Мои Анализы»

## 1. Общее описание проекта

«Мои Анализы» — облачный веб-сервис и мобильное приложение для автоматического распознавания, структурирования и предварительной интерпретации медицинских лабораторных анализов. Пользователь загружает файл с результатами анализов, система извлекает показатели, сравнивает их с референсными значениями, подсвечивает отклонения, формирует понятное резюме и рекомендации по дальнейшим действиям.

Сервис не заменяет врача и не должен позиционироваться как инструмент постановки окончательного диагноза. Все результаты AI-интерпретации должны сопровождаться медицинским дисклеймером и рекомендацией обратиться к профильному специалисту.

### 1.1 Цели

- Сократить время ручной расшифровки лабораторных анализов.
- Дать пользователю понятное объяснение отклонений без медицинской терминологической перегрузки.
- Создать защищенное хранилище истории анализов и динамики показателей.
- Подготовить архитектуру для масштабирования на B2C, B2B и лабораторные интеграции.

### 1.2 Задачи

- Реализовать загрузку PDF, JPEG, PNG, WebP, HEIC до 20 МБ.
- Выполнить OCR и нормализацию медицинских показателей.
- Сопоставить показатели с референсами с учетом пола, возраста, единиц измерения и лабораторных диапазонов.
- Сформировать предварительную интерпретацию, возможные причины отклонений и рекомендации.
- Обеспечить хранение истории анализов, результатов распознавания и версий интерпретаций.
- Реализовать административную панель для управления пользователями, справочниками, контентом и инцидентами.
- Обеспечить безопасность персональных и медицинских данных.

### 1.3 Целевая аудитория

- Пациенты, которые хотят быстро понять результаты анализов.
- Врачи и консультанты, которым нужен быстрый предварительный разбор входящих анализов.
- Участники лабораторий и медицинских организаций, которым требуется клиентский цифровой кабинет.
- Администраторы сервиса, медицинские редакторы и специалисты поддержки.

### 1.4 Референс

В качестве функционального ориентира использовать сервис moyanaliz.ru и аналогичные решения для расшифровки анализов. Для проектирования MVP необходимо заложить минимум следующие пользовательские сценарии:

- загрузка результата анализа;
- автоматическое выделение показателей;
- отображение нормы, фактического значения и отклонения;
- текстовая интерпретация результата;
- история загруженных анализов;
- понятный интерфейс на мобильных устройствах.

Перед разработкой рекомендуется провести отдельный конкурентный анализ moyanaliz.ru: онбординг, ограничения загрузки, формат результата, монетизация, юридические дисклеймеры, скорость обработки, UX мобильной версии.

## 2. Роли пользователей

### 2.1 Незарегистрированный пользователь

- Просматривает публичные страницы: главная, возможности, тарифы, FAQ, правовые документы.
- Может загрузить один демонстрационный анализ с ограниченным результатом, если это разрешено бизнес-моделью.
- Получает предложение зарегистрироваться для сохранения истории и полного отчета.
- Может зарегистрироваться через email, OAuth или номер телефона, если канал подтверждения включен.

### 2.2 Авторизованный пользователь

- Управляет профилем и медицинскими параметрами: пол, дата рождения, вес, рост, беременность, хронические состояния, принимаемые препараты.
- Загружает файлы анализов перетаскиванием или через файловый диалог.
- Видит статус обработки: загружено, идет OCR, идет AI-анализ, требуется проверка, готово, ошибка.
- Получает структурированный отчет по анализу.
- Просматривает историю анализов и динамику показателей.
- Добавляет ручные комментарии к анализам.
- Исправляет распознанные значения, если OCR ошибся.
- Экспортирует отчет в PDF.
- Удаляет анализы и аккаунт в рамках политики хранения данных.
- Управляет согласием на обработку персональных и медицинских данных.

### 2.3 Администратор

- Управляет пользователями, ролями и блокировками.
- Просматривает журнал технических событий без раскрытия медицинских данных сверх необходимых прав.
- Управляет справочниками показателей, единиц измерения, синонимов и референсных диапазонов.
- Управляет контентом: FAQ, дисклеймеры, медицинские статьи, шаблоны рекомендаций.
- Просматривает очередь проблемных распознаваний.
- Запускает повторную обработку анализа.
- Управляет тарифами, лимитами и промокодами, если монетизация включена.
- Настраивает интеграции с OCR, AI, платежами, S3-хранилищем и уведомлениями.

### 2.4 Медицинский редактор

- Проверяет шаблоны интерпретаций и справочники показателей.
- Добавляет медицинские предупреждения и ограничения.
- Маркирует рискованные темы, где AI должен давать только общую рекомендацию обратиться к врачу.
- Просматривает обезличенные примеры ошибок распознавания и интерпретации.

## 3. Функциональные требования

### 3.1 Регистрация и авторизация

- Email и пароль с подтверждением email.
- OAuth: Google, Apple, Yandex при необходимости рынка РФ.
- Восстановление пароля.
- Двухфакторная аутентификация для администраторов.
- JWT или сессионная авторизация с refresh token rotation.
- Поддержка удаления аккаунта и выгрузки персональных данных.

### 3.2 Загрузка файлов

- Поддерживаемые форматы: PDF, JPEG, PNG, WebP, HEIC.
- Максимальный размер файла: 20 МБ.
- Поддержка drag-and-drop.
- Множественная загрузка в рамках лимитов тарифа.
- Предпросмотр изображений и PDF.
- Проверка MIME-типа и расширения.
- Антивирусная проверка файлов.
- Удаление EXIF-метаданных из изображений перед обработкой.
- Хранение оригинала в зашифрованном объектном хранилище.

### 3.3 OCR и извлечение данных

- OCR для изображений и PDF-сканов.
- Извлечение текста из текстового PDF без OCR, если возможно.
- Распознавание таблиц, единиц измерения, референсных интервалов, флагов H/L.
- Нормализация названий показателей по справочнику синонимов.
- Приведение единиц измерения к каноническому виду, если конвертация безопасна.
- Сохранение confidence score по каждому распознанному полю.
- Передача низкоуверенных результатов в режим ручной проверки или пользовательского подтверждения.

### 3.4 AI-интерпретация

- AI-модель должна получать структурированные показатели, пользовательские параметры и ограниченный контекст, а не полный сырой файл без необходимости.
- Результат AI должен возвращаться в строгой JSON-схеме.
- В отчете должны быть:
  - краткое резюме;
  - список отклонений;
  - возможные причины;
  - рекомендации по обращению к врачу;
  - показатели в норме;
  - предупреждения о критических значениях;
  - уровень уверенности;
  - дисклеймер.
- Запрещено формулировать окончательный диагноз как медицинский факт.
- Для критических значений показывать срочное предупреждение обратиться за медицинской помощью.
- Хранить версию модели, prompt template и версию справочников, использованных для отчета.

### 3.5 Отчет пользователя

- Отображение анализа в виде таблицы: показатель, значение, единица, норма, статус, комментарий.
- Цветовые статусы: ниже нормы, выше нормы, критично, норма, не распознано.
- Группировка показателей по системам: кровь, печень, почки, гормоны, витамины, воспаление и др.
- Графики динамики по повторным анализам.
- Экспорт отчета в PDF.
- Возможность исправить распознанное значение и пересчитать интерпретацию.
- Отображение первоисточника: ссылка на оригинальный файл и OCR-текст.

### 3.6 История анализов

- Список анализов с фильтрами по дате, типу, статусу, отклонениям.
- Карточка анализа с исходным файлом, распознанными данными и отчетом.
- Сравнение нескольких анализов одного типа.
- Удаление анализа пользователем.
- Политика soft delete и последующего физического удаления по расписанию.

### 3.7 Уведомления

- Email-уведомление о готовности отчета.
- Push-уведомления в PWA или мобильном приложении.
- Уведомления об ошибке обработки и необходимости повторной загрузки.
- Настройки подписки на уведомления.

### 3.8 Админ-панель

- Dashboard с метриками: загрузки, ошибки OCR, ошибки AI, активные пользователи, среднее время обработки.
- Управление пользователями и ролями.
- Управление справочниками показателей.
- Просмотр очередей обработки.
- Просмотр технических логов и audit trail.
- Управление текстами дисклеймеров, FAQ и статей.
- Управление тарифами и лимитами.

### 3.9 Платежи и тарифы

MVP может стартовать без платежей, но архитектура должна поддерживать тарифы:

- бесплатный тариф с ограничением количества анализов;
- разовая покупка отчета;
- месячная подписка;
- B2B-доступ для клиник и лабораторий;
- лимиты по количеству файлов, размеру хранилища и частоте AI-запросов.

## 4. Нефункциональные требования

### 4.1 Безопасность

- Шифрование данных при передаче: TLS 1.2+.
- Шифрование данных в покое: AES-256 или managed encryption облачного провайдера.
- Раздельное хранение файлов, персональных данных и технических логов.
- Принцип минимальных прав для сервисных аккаунтов.
- Role-Based Access Control для админ-панели.
- Audit log для действий с медицинскими данными.
- Secret management через облачный Secret Manager или Vault.
- Регулярная ротация ключей и токенов.
- Защита от CSRF, XSS, SSRF, IDOR, SQL injection.
- Rate limiting и защита от brute force.
- Антивирусная проверка загружаемых файлов.
- Маскирование чувствительных данных в логах.

### 4.2 Соответствие требованиям

- Для РФ: учитывать 152-ФЗ, требования к согласию на обработку персональных данных, локализацию персональных данных при необходимости.
- Для США: HIPAA применим только при работе с covered entities/business associates; если сервис выходит на этот рынок, нужен отдельный compliance-проект.
- Для ЕС: GDPR, право на удаление, выгрузку и ограничение обработки данных.
- Перед запуском обязательна юридическая экспертиза пользовательского соглашения, политики конфиденциальности, медицинских дисклеймеров и согласий.

### 4.3 Производительность

- SLA целевого production-сервиса: 99,9% после стабилизации.
- Время загрузки основного интерфейса: до 2 секунд при нормальном соединении.
- Время обработки типового анализа: до 2 минут для 95-го перцентиля.
- Асинхронная обработка файлов через очередь задач.
- Горизонтальное масштабирование воркеров OCR и AI.
- Кеширование справочников и публичного контента.

### 4.4 Масштабируемость

- Stateless API-сервисы.
- Очередь задач для OCR, AI и PDF-экспорта.
- Отдельные воркеры для тяжелой обработки.
- S3-совместимое хранилище для файлов.
- PostgreSQL как основная транзакционная база.
- Redis для кеша, rate limiting и краткоживущих задач.
- Возможность подключения новых AI/OCR-провайдеров через adapter interface.

### 4.5 Надежность

- Повторные попытки для временных ошибок OCR/AI.
- Dead-letter queue для неуспешных задач.
- Idempotency keys для операций загрузки и обработки.
- Резервное копирование PostgreSQL.
- Lifecycle policy для файлов в объектном хранилище.
- Monitoring, alerting и incident response process.

### 4.6 Адаптивность и доступность

- Mobile-first интерфейс.
- PWA: установка на устройство, push-уведомления, offline-shell для истории без чувствительных данных.
- WCAG 2.1 AA как целевой уровень доступности.
- Поддержка темной темы при наличии в дизайн-системе.
- Корректная работа на iOS Safari, Android Chrome, desktop Chrome, Safari, Firefox.

## 5. Архитектура системы

### 5.1 Рекомендуемый стек

Для быстрого MVP рекомендуется Next.js/React или Nuxt 3 с TypeScript. Так как исходный референс указывает на nuxt-ui-templates/dashboard, базовый вариант:

- Frontend: Nuxt 3, TypeScript, TailwindCSS, Nuxt UI.
- Backend: NestJS или FastAPI.
- Database: PostgreSQL.
- Cache/Queue: Redis + BullMQ для Node.js или Celery/RQ для Python.
- Object Storage: Cloudflare R2, AWS S3, Selectel S3 или Vercel Blob.
- OCR: Google Cloud Vision, AWS Textract, Azure Document Intelligence или Tesseract как fallback.
- AI: OpenAI API или Google Gemini через абстрактный AI provider layer.
- Monitoring: Sentry, OpenTelemetry, Prometheus/Grafana при необходимости.
- CI/CD: GitHub Actions.
- Hosting: Vercel для frontend, Render/Fly.io/AWS/GCP для backend и workers; для данных РФ рассмотреть российские облака.

### 5.2 Выбор backend

NestJS подходит, если команда ориентирована на TypeScript и хочет единый язык на frontend/backend. FastAPI подходит, если команда планирует активную ML/OCR-обработку на Python. Для MVP предпочтителен NestJS при TypeScript-команде, с отдельным Python microservice только при появлении сложной ML-логики.

### 5.3 Компоненты

- Web App: пользовательский кабинет и админ-панель.
- API Gateway / Backend API: авторизация, файлы, анализы, пользователи, тарифы.
- Processing Workers: OCR, нормализация, AI-интерпретация, PDF-экспорт.
- Database: пользователи, анализы, результаты, справочники, audit logs.
- Object Storage: оригиналы файлов, OCR-артефакты, экспортированные отчеты.
- Queue: фоновые задачи обработки.
- AI Provider Adapter: OpenAI, Gemini, локальная модель.
- OCR Provider Adapter: Textract, Vision, Azure, Tesseract.
- Admin Console: управление и модерация.

### 5.4 Поток обработки анализа

1. Пользователь загружает файл.
2. Backend проверяет формат, размер, MIME, вирусы и права пользователя.
3. Файл сохраняется в S3-compatible storage с шифрованием.
4. Создается запись анализа в PostgreSQL со статусом `uploaded`.
5. В очередь ставится задача OCR.
6. OCR worker извлекает текст и таблицы, сохраняет сырой результат.
7. Normalization worker выделяет показатели, единицы, нормы и confidence score.
8. AI worker формирует интерпретацию по строгой JSON-схеме.
9. Backend сохраняет отчет, меняет статус на `completed` или `needs_review`.
10. Пользователь получает уведомление и открывает отчет.

### 5.5 Основные сущности данных

- User: аккаунт, роль, настройки безопасности.
- UserProfile: медицинские параметры и согласия.
- Analysis: загруженный анализ, статус, источник, даты.
- AnalysisFile: оригинальный файл и производные артефакты.
- ExtractedMarker: показатель, значение, единица, норма, confidence.
- MarkerDictionary: канонические показатели, синонимы, группы.
- ReferenceRange: нормы по полу, возрасту, состояниям и единицам.
- Interpretation: AI-отчет, версия модели, версия prompt, JSON-результат.
- AuditLog: события доступа и изменений.
- Subscription: тариф, лимиты, платежный статус.

## 6. AI и медицинская безопасность

### 6.1 Рекомендуемый AI-подход

- Использовать AI не для OCR как единственного источника истины, а для интерпретации уже структурированных данных.
- Применять deterministic validation после ответа AI: JSON Schema, проверка обязательных полей, запрет неподдержанных диагнозов.
- Разделить medical knowledge base и prompt templates.
- Хранить версии prompt и модели для воспроизводимости.
- Добавить human review workflow для низкой уверенности, критических значений и спорных результатов.

### 6.2 Ограничения AI

- Не выдавать окончательные диагнозы.
- Не назначать лекарства и дозировки.
- Не отменять назначения врача.
- Не интерпретировать анализ вне контекста симптомов как медицинский факт.
- Всегда указывать, что результат является предварительной информационной расшифровкой.

### 6.3 Критические значения

Для критических значений система должна:

- показывать заметное предупреждение;
- рекомендовать срочно связаться с врачом или экстренной помощью;
- не скрывать предупреждение внутри длинного AI-текста;
- логировать срабатывание правила без раскрытия лишних данных в технических логах.

### 7.3 Состояния интерфейса

- Empty state для первой загрузки.
- Loading/progress state при обработке.
- Error state с понятной причиной и действием.
- Needs review state при низкой уверенности OCR.
- Completed state с отчетом.
- Deleted/expired state для удаленных файлов.

Для подготовки мобильного релиза требуется отдельный набор промо-скриншотов в текущем стиле продукта. Полные требования зафиксированы в `docs/store-screenshots-spec.md`.

Обязательные наборы:

## 12. Критерии приемки MVP

- Пользователь может зарегистрироваться, загрузить поддерживаемый файл до 20 МБ и получить отчет.
- Система сохраняет оригинальный файл, OCR-текст, структурированные показатели и AI-интерпретацию.
- Пользователь видит отклонения, нормы, рекомендации и дисклеймер.
- Пользователь может открыть историю анализов и удалить анализ.
- Администратор может управлять пользователями и справочником показателей.
- Все медицинские данные передаются по TLS и хранятся с шифрованием.
- В логах отсутствуют несанкционированные персональные и медицинские данные.
- Есть audit log доступа к чувствительным данным.
- Основные сценарии покрыты автоматическими тестами.
- Production окружение имеет мониторинг, алерты и резервное копирование.

## 13. Definition of Done

- Код проходит lint, typecheck, unit и integration tests.
- Ключевые пользовательские сценарии покрыты E2E-тестами.
- Обновлена техническая документация и API-документация.
- Обновлены правовые тексты и дисклеймеры.
- Проведен security review для затронутых компонентов.
- Новые env-переменные описаны в `.env.example`.
- Все миграции БД обратимы или имеют понятный rollback plan.
- PR содержит описание изменений, тестов, рисков и скриншоты для UI-изменений.
