# Реализация

Документ фиксирует поэтапную реализацию проекта «Мои Анализы» по техническому заданию из `README.md`. При закрытии каждого issue сюда добавляется краткий результат: что сделано, какие артефакты появились, как проверено и что остается следующим шагом.

## Правила ведения

- Обновлять документ в каждом PR, который закрывает issue или меняет план реализации.
- Фиксировать дату, номер issue, ссылку на PR и проверку результата.
- Отмечать только фактически выполненные работы, не подменяя план отчетом о готовности.
- Если issue выявляет новый риск, зависимость или открытый вопрос, добавлять его в соответствующий раздел.

## Текущий статус

Проект находится на этапе frontend MVP-прототипа. Базовая архитектура, окружение и статический dashboard-прототип приложения созданы; реальная авторизация, API, хранение файлов, OCR и AI-обработка остаются следующими этапами.

## Поэтапный план работ

### Этап 0. Аналитика и проектирование

Цель: превратить техническое задание в проверенный набор продуктовых, юридических и архитектурных решений.

Ключевые задачи:

- Провести конкурентный анализ moyanaliz.ru и 3-5 аналогичных сервисов.
- Определить рынок первого запуска: РФ, ЕС, США или международная модель.
- Согласовать MVP-сценарии, ограничения бесплатного режима и монетизацию.
- Описать пользовательские персоны, CJM и основные user flows.
- Подготовить прототипы ключевых экранов: загрузка, обработка, отчет, история, профиль, админ-панель.
- Утвердить стек, провайдеров OCR/AI, модель хранения медицинских данных и compliance-подход.

Результат этапа:

- Product brief и MVP scope.
- Competitive analysis.
- Legal/compliance checklist.
- Архитектурное решение и технический план MVP.
- Прототипы и acceptance criteria для первых пользовательских сценариев.

### Этап 1. Каркас проекта и окружение

Цель: создать базу разработки, CI/CD и локальное окружение.

Ключевые задачи:

- Создать структуру приложения: frontend, backend, shared packages, infra, docs.
- Настроить TypeScript, linting, formatting, тестовый фреймворк и pre-commit проверки.
- Подготовить Docker Compose для PostgreSQL, Redis и S3-compatible storage.
- Добавить `.env.example`, секреты для dev/staging и инструкции запуска.
- Настроить GitHub Actions для lint, typecheck и tests.
- Завести базовые health checks и структуру логирования.

Результат этапа:

- Репозиторий готов к разработке приложения.
- Разработчик может поднять локальное окружение по инструкции.
- CI проверяет базовое качество кода.

### Этап 2. Frontend MVP-прототип

Цель: реализовать пользовательский dashboard без реальной обработки анализов, но с полноценными состояниями интерфейса.

Ключевые задачи:

- Собрать layout пользовательского кабинета и адаптивную навигацию.
- Реализовать auth screens и профиль пользователя в mock-режиме.
- Реализовать upload UI с drag-and-drop, проверкой формата и лимита 20 МБ.
- Собрать экраны статуса обработки: uploaded, OCR, AI, needs_review, completed, error.
- Собрать mock-отчет с таблицей показателей, отклонениями, дисклеймером и рекомендациями.
- Реализовать историю анализов и карточку анализа на mock-данных.
- Подготовить базовые экраны админ-панели.

Результат этапа:

- Интерактивный frontend-прототип основных сценариев.
- Скриншоты или preview-ссылка для приемки UX.
- Начальные E2E/smoke tests для ключевых экранов.

### Этап 3. Backend MVP и данные

Цель: реализовать API, авторизацию, хранение файлов и доменную модель анализов.

Ключевые задачи:

- Реализовать регистрацию, вход, refresh token flow и роли.
- Реализовать профиль пользователя, согласия и удаление аккаунта.
- Реализовать загрузку файлов с проверкой MIME, расширения и размера.
- Сохранять оригиналы файлов в объектное хранилище, а метаданные в PostgreSQL.
- Добавить сущности Analysis, AnalysisFile, ExtractedMarker, Interpretation, AuditLog.
- Реализовать историю, карточку анализа, удаление и повторную обработку.
- Реализовать базовые админские API для пользователей и анализов.

Результат этапа:

- Backend API поддерживает основные пользовательские сценарии без OCR/AI.
- Есть миграции БД, тесты API и документация эндпоинтов.
- Audit log фиксирует чувствительные действия.

### Этап 4. OCR, нормализация и AI-интерпретация

Цель: подключить асинхронную обработку файлов и формирование предварительного отчета.

Ключевые задачи:

- Настроить очередь задач и воркеры OCR, нормализации, AI и PDF-экспорта.
- Подключить выбранного OCR-провайдера через adapter interface.
- Извлекать текст из текстовых PDF без OCR, когда это возможно.
- Нормализовать показатели, единицы, нормы, флаги H/L и confidence score.
- Создать справочник показателей, синонимов и референсных диапазонов.
- Подключить AI provider adapter и строгую JSON Schema validation.
- Добавить правила для критических значений и обязательный медицинский дисклеймер.
- Реализовать пользовательские исправления и повторный расчет интерпретации.

Результат этапа:

- Пользователь получает структурированный отчет по загруженному анализу.
- Низкоуверенные и критические случаи корректно выделяются.
- Хранятся версии модели, prompt template и справочников.

### Этап 5. Безопасность и соответствие требованиям

Цель: подготовить MVP к работе с персональными и медицинскими данными.

Ключевые задачи:

- Реализовать RBAC, 2FA для администраторов и принцип минимальных прав.
- Настроить secure headers, CSRF/XSS/SSRF/IDOR protections и rate limiting.
- Добавить антивирусную проверку загрузок и очистку EXIF-метаданных.
- Обеспечить шифрование данных при передаче и в покое.
- Добавить маскирование чувствительных данных в логах.
- Провести threat modeling и security review.
- Подготовить правовые тексты: политика конфиденциальности, согласия, дисклеймеры.

Результат этапа:

- Сформирован минимальный security baseline.
- Зафиксированы юридические ограничения запуска.
- Критические действия покрыты аудитом и проверками.

### Этап 6. QA, beta и запуск

Цель: стабилизировать продукт перед публичным MVP.

Ключевые задачи:

- Подготовить набор обезличенных тестовых анализов.
- Покрыть pipeline unit, integration и E2E тестами.
- Провести нагрузочное тестирование очередей OCR/AI.
- Проверить восстановление после ошибок внешних провайдеров.
- Провести UAT с медицинским редактором.
- Настроить monitoring, alerting, backups и incident process.
- Запустить закрытую beta-группу, исправить критические дефекты и подготовить публичный MVP.

Результат этапа:

- MVP готов к ограниченному production-запуску.
- Есть мониторинг, алерты, резервное копирование и процесс поддержки.

## Ближайшие issue

Рекомендуемый порядок следующих задач:

1. Начать с исправления и приемки Этапа 2: уточнить мобильные экраны, состояния оценки риска, чекапов, напоминаний и AI-чата.
2. Сверить новые экраны с `docs/store-screenshots-spec.md` и `docs/план работ исправления ошибок.md`.
3. Проверить OCR-провайдеров на русскоязычных лабораторных бланках и выбрать первый adapter.
4. Уточнить AI/OCR JSON-контракты для risk summary, checkups, medication reminders и agent chat actions.
5. Подготовить юридические тексты для экспертной проверки.
6. Провести mobile/PWA приемку и отобрать финальные витринные скриншоты.
