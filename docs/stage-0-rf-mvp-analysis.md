# Этап 0: аналитика и область MVP для РФ

Дата: 2026-05-12  
Issue: https://github.com/glsfull/med-analiz/issues/5  
PR: https://github.com/glsfull/med-analiz/pull/6

## 1. Решение по рынку первого запуска

Первый запуск сервиса «Мои Анализы» планируется только для рынка РФ.

Практические следствия:

- интерфейс, юридические тексты, справочники, поддержка и медицинские дисклеймеры готовятся на русском языке;
- в MVP учитываются требования 152-ФЗ, согласие на обработку персональных данных и отдельное согласие на обработку медицинских данных;
- персональные данные и медицинские файлы должны храниться у провайдера, подходящего для запуска в РФ;
- OAuth-провайдеры для первого запуска: email как обязательный канал, Yandex ID как приоритетная опция, Google и Apple как необязательные расширения;
- платежи, если будут включены в MVP, проектируются под РФ-эквайринг и рублевые тарифы;
- сценарии ЕС, США, HIPAA и GDPR остаются архитектурными ограничениями на будущее, но не входят в запуск первой версии.

## 2. Рабочая гипотеза продукта

Сервис должен закрыть сценарий пользователя, который получил лабораторный бланк и хочет быстро понять:

- какие показатели вышли за пределы нормы;
- насколько отклонения существенны;
- какие причины могут быть связаны с отклонениями;
- к какому врачу или виду консультации стоит обратиться;
- что изменилось по сравнению с предыдущими анализами.

AI-интерпретация остается предварительной и не заменяет врача. Все отчеты должны показывать дисклеймер до медицинских выводов и рядом с итоговым резюме.

## 3. Конкурентный анализ

На этапе проектирования нужно проверить moyanaliz.ru и близкие сервисы по единой матрице. Цель не в копировании интерфейса, а в фиксации ожиданий рынка и обязательных ограничений.

### 3.1 Объекты анализа

- moyanaliz.ru как основной функциональный ориентир из ТЗ;
- российские сервисы расшифровки анализов и медицинские справочники с интерпретацией показателей;
- личные кабинеты лабораторий РФ, если они показывают динамику, нормы и пояснения;
- международные аналоги только как источник UX-паттернов, без переноса юридических и платежных решений в MVP.

### 3.2 Матрица сравнения

| Критерий | Что фиксировать | Значение для MVP |
| --- | --- | --- |
| Онбординг | Нужна ли регистрация до загрузки, какие согласия показываются | Решить, разрешать ли демо-загрузку без аккаунта |
| Форматы загрузки | PDF, фото, сканы, ограничения размера | Подтвердить лимит 20 МБ и список форматов из ТЗ |
| Скорость обработки | Обещанное и фактическое время до результата | Задать ожидание до 2 минут для типового анализа |
| Качество результата | Таблица, отклонения, объяснения, рекомендации | Определить минимальную структуру отчета |
| Дисклеймеры | Где и как раскрывается медицинское ограничение | Сделать дисклеймер обязательной частью отчета |
| Монетизация | Бесплатный лимит, разовая покупка, подписка | Выбрать простую модель для MVP |
| История | Есть ли сохранение и сравнение анализов | Включить историю в авторизованный сценарий |
| Мобильный UX | Удобство загрузки фото и чтения отчета | Проектировать mobile-first для загрузки и отчета |
| Исправления OCR | Можно ли править распознанные значения | Оставить исправления в MVP, если позволяет срок |
| Поддержка | Что происходит при ошибке распознавания | Нужны статусы ошибки и повторной обработки |

### 3.3 Предварительные выводы для проектирования

- Главная ценность MVP: не загрузка файла сама по себе, а понятный отчет с выделением отклонений и безопасными следующими шагами.
- Демо без регистрации полезно для конверсии, но в РФ оно усложняет согласия и хранение медицинских данных. Для первой версии безопаснее требовать аккаунт и согласия до полной обработки.
- В отчете важнее показать таблицу показателей, отклонения и дисклеймер, чем добавлять расширенные статьи и длинные медицинские объяснения.
- История анализов нужна в MVP, потому что без нее сервис слабо отличается от одноразовой расшифровки.
- Монетизацию лучше не включать в первый технический MVP, но оставить тарифные лимиты в модели данных.

## 4. MVP scope

### 4.1 Входит в MVP

- регистрация и вход по email;
- профиль пользователя с базовыми медицинскими параметрами: пол, дата рождения, беременность, хронические состояния и принимаемые препараты как текстовые поля;
- обязательные согласия на обработку персональных и медицинских данных;
- загрузка одного или нескольких файлов анализа в форматах PDF, JPEG, PNG, WebP, HEIC до 20 МБ;
- статусы обработки: uploaded, ocr_processing, ai_processing, needs_review, completed, failed;
- извлечение текста из текстовых PDF и OCR для изображений/сканов через провайдера-адаптер;
- нормализация показателей, единиц, референсов и флагов отклонений;
- отчет с таблицей показателей, кратким резюме, отклонениями, возможными причинами, рекомендацией по обращению к врачу и дисклеймером;
- история анализов, карточка анализа и soft delete;
- ручное исправление распознанных значений с пересчетом статусов;
- базовая админ-панель для просмотра очереди проблемных обработок и управления справочником показателей;
- audit log для действий с медицинскими данными;
- экспорт отчета в PDF как желательная, но не блокирующая функция MVP.

### 4.2 Не входит в MVP

- мобильные приложения в App Store и Google Play;
- B2B-кабинеты клиник и лабораторий;
- интеграции с лабораторными информационными системами;
- автоматическая постановка диагнозов;
- телемедицина и запись к врачу;
- поддержка рынков ЕС и США;
- HIPAA/GDPR-процессы как полноценный compliance-проект;
- продвинутая BI-аналитика, промокоды и сложная подписочная модель;
- медицинские статьи и персонализированные лечебные планы.

## 5. Пользовательские сценарии первого релиза

### 5.1 Первый анализ

1. Пользователь создает аккаунт.
2. Заполняет минимальный профиль.
3. Принимает согласия.
4. Загружает файл анализа.
5. Видит статус обработки.
6. Получает отчет.
7. Открывает таблицу отклонений и рекомендации.

### 5.2 Исправление распознавания

1. Пользователь открывает готовый отчет.
2. Видит показатель с низкой уверенностью или ошибочным значением.
3. Исправляет значение, единицу или референс.
4. Система пересчитывает статус показателя.
5. AI-интерпретация помечается как требующая повторного расчета или пересчитывается.

### 5.3 История и динамика

1. Пользователь открывает историю анализов.
2. Фильтрует записи по дате или типу анализа.
3. Открывает карточку анализа.
4. Сравнивает повторяющийся показатель с прошлым результатом.

### 5.4 Ошибка обработки

1. Пользователь загружает файл.
2. OCR или AI-этап завершается ошибкой.
3. Сервис показывает понятную причину и безопасное действие: повторить загрузку, выбрать другой файл или отправить на ручную проверку.
4. Технические детали сохраняются в журнале без раскрытия лишних медицинских данных.

## 6. Legal и compliance checklist для РФ

До публичного запуска нужно подготовить и проверить с юристом:

- пользовательское соглашение;
- политику конфиденциальности;
- согласие на обработку персональных данных;
- отдельное согласие на обработку специальных категорий персональных данных, включая медицинские данные;
- медицинский дисклеймер о предварительном характере интерпретации;
- правила удаления аккаунта, анализа и файлов;
- сроки хранения оригиналов, OCR-текста, структурированных показателей и AI-интерпретаций;
- порядок обработки обращений пользователя по данным;
- модель размещения данных у инфраструктурного провайдера;
- требования к журналированию доступа сотрудников к медицинским данным.

## 7. Архитектурные решения для MVP

- Основное приложение: web dashboard с адаптивным интерфейсом.
- API: stateless backend с ролевой моделью пользователя, администратора и медицинского редактора.
- Обработка файлов: асинхронная очередь для OCR, нормализации, AI-интерпретации и PDF-экспорта.
- Хранилище файлов: S3-compatible object storage с шифрованием и раздельными bucket/prefix для оригиналов и производных артефактов.
- База данных: PostgreSQL для пользователей, анализов, показателей, интерпретаций, согласий и audit log.
- Кеш/очередь: Redis или совместимый брокер для задач MVP.
- OCR и AI: adapter interfaces, чтобы можно было заменить провайдера без переписывания доменной логики.
- AI-ответ: строгая JSON Schema validation, хранение версии модели, prompt template и версии справочников.

## 8. Acceptance criteria для закрытия Этапа 0

Этап 0 можно считать завершенным, когда:

- подтвержден RF-only scope первого запуска;
- конкурентный анализ заполнен по фактическим данным минимум по moyanaliz.ru и 3 аналогам;
- утвержден список MVP-сценариев и список исключений из MVP;
- подготовлены low-fidelity прототипы ключевых экранов;
- выбран технологический стек и провайдеры OCR/AI для первого прототипа;
- юридический checklist передан на экспертную проверку;
- для Этапа 1 заведены задачи по каркасу проекта, окружению и CI.

## 9. Открытые вопросы

- Разрешать ли демо-загрузку без регистрации, если полный отчет все равно доступен только после аккаунта и согласий?
- Включать ли PDF-экспорт в обязательный MVP или оставить на следующий релиз?
- Нужна ли ручная медицинская модерация интерпретаций на beta-этапе?
- Какой OCR-провайдер лучше подходит под документы на русском языке и требования хранения данных?
- Будет ли монетизация включена в публичный MVP или первый запуск останется бесплатной beta-версией?
- Какие лабораторные панели покрывать первыми: ОАК, биохимия, гормоны, витамины, моча?
